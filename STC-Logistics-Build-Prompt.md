## 1\. Project Summary

Build **STC Logistics**, a freight forwarding website for shipping goods from China to Africa by air and sea. Design and UX should be inspired by [https://www.fycargo.com](https://www.fycargo.com) (warehouse-code model, prominent tracking search bar, "how it works" flow, service pages for air/sea freight, customer dashboard) — but with original branding, copy, and layout, not a copy of their code or assets.

**Stack:**

- **Frontend/Backend:** Next.js 14+ (App Router), TypeScript, Tailwind CSS, deployed on **Vercel**  
- **Database/Auth/Storage:** **Supabase** (Postgres, Auth, Storage, Row Level Security)  
- **External API:** Maersk Track & Trace API (DCSA v2.2 standard) for live sea-freight tracking  
- **Hosting of secrets:** Vercel Environment Variables (never client-exposed)

---

## 2\. Core Business Logic

STC Logistics operates a China-based consolidation warehouse. The flow has two stages:

### Stage 1 — Pre-Alert (China local leg)

1. On signup, every customer is automatically issued a unique **Customer Warehouse Code** (e.g. `STC-CN-0001`), generated by the system attached with the company’s forwarding warehouse in China. This is the address a customer must give to their supplier when buying.  
2. When a customer buys goods from a Chinese supplier, they get a **local China courier tracking number** (e.g. from SF Express, YTO, etc.) from the supplier.  
3. The customer logs into the site and submits a **Pre-Alert**: local tracking number \+ their warehouse code (auto-filled) \+ item description, quantity, estimated value, supplier name.  
4. Warehouse staff (internal role) see incoming pre-alerts in an admin queue. When a parcel physically arrives at the China warehouse, staff search by local tracking number or warehouse code and mark it **"Arrived at Warehouse"**, optionally logging weight/dimensions/photos.

### Stage 2 — International Shipment

5. Once one or more parcels are marked "Arrived at Warehouse," the customer can select them in their dashboard and submit a **Shipment Request**: choose destination country (Africa), choose mode (air or sea), and optionally consolidate multiple parcels into one shipment.  
6. Staff/admin review the request, calculate/enter freight cost, and once the shipment is booked, the admin creates a **Shipment record** and:  
   - Generates an **internal STC tracking number** (e.g. `STC-SH-2026-000123`) — this is the only tracking number the customer ever needs to use on the site.  
   - If sea freight and booked with Maersk, admin attaches the Maersk **carrierBookingReference** and/or **transportDocumentReference** to the shipment record.  
7. The customer tracks their shipment **only on the STC website** by entering the internal STC tracking number in the site's tracking widget. The site looks up the linked Maersk reference server-side, calls the Maersk API, and renders a timeline — the customer never sees a Maersk URL, reference format, or is redirected off-site.  
8. Air freight shipments (not covered by Maersk ocean tracking) use manual milestone updates entered by staff (see §6 fallback).

## 3\. Maersk API Integration Details

Reference spec: DCSA Track & Trace Events API v1.5.0 (`GET /events`), OAuth2 \+ `Consumer-Key` header.

- **Base URL (prod):** `https://api.maersk.com/track-and-trace-private`  
- **Auth:** OAuth 2.0 Bearer token \+ `Consumer-Key` header (client credentials flow). Store `MAERSK_CONSUMER_KEY`, `MAERSK_CONSUMER_SECRET`, `MAERSK_CLIENT_ID` as Vercel env vars — call Maersk **only from server-side API routes / Edge functions**, never from the browser.  
- **Query params to use:** at least one of `carrierBookingReference`, `transportDocumentReference`, or `equipmentReference` (whichever the admin recorded), plus `eventType=SHIPMENT,TRANSPORT,EQUIPMENT` to get the full timeline.  
- **Rate limit:** 4000 calls/hour, 60/min per Consumer Key — so **cache results in Supabase** (see `tracking_events` table below) rather than calling Maersk on every page view. Refresh on a schedule (e.g. every 30–60 min via a Vercel Cron job) or on-demand with a short cache TTL (e.g. 15 min) when a customer opens the tracking page.  
- **Known constraint to design around:** Maersk returns **404** if the Consumer Key's registered Maersk Customer Code is not a party to that shipment. This means live Maersk data will only work for bookings made directly under STC Logistics' (or its Maersk-registered partner's) account. Build the tracking system so it **gracefully falls back to manually-entered milestones** (see §6) when Maersk returns 404/no data, rather than showing an error to the customer.

## 4\. Supabase Data Model

\-- Roles: customer, warehouse\_staff, admin

create type user\_role as enum ('customer', 'warehouse\_staff', 'admin');

create table profiles (

  id uuid primary key references auth.users(id),

  warehouse\_code text unique not null, \-- e.g. STC-CN-0001, generated on signup

  full\_name text,

  phone text,

  country text,

  role user\_role not null default 'customer',

  created\_at timestamptz default now()

);

create table parcels ( \-- Stage 1: pre-alerts / warehouse intake

  id uuid primary key default gen\_random\_uuid(),

  customer\_id uuid references profiles(id) not null,

  local\_tracking\_number text not null,

  supplier\_name text,

  item\_description text,

  quantity int default 1,

  declared\_value numeric,

  status text not null default 'pending' check (status in

    ('pending','arrived','flagged','consolidated','cancelled')),

  weight\_kg numeric,

  dimensions text,

  photos text\[\], \-- Supabase Storage URLs

  arrived\_at timestamptz,

  created\_at timestamptz default now()

);

create table shipments ( \-- Stage 2: international leg

  id uuid primary key default gen\_random\_uuid(),

  customer\_id uuid references profiles(id) not null,

  stc\_tracking\_number text unique not null, \-- e.g. STC-SH-2026-000123

  mode text not null check (mode in ('air','sea')),

  destination\_country text not null,

  status text not null default 'processing' check (status in

    ('processing','booked','in\_transit','customs','out\_for\_delivery','delivered','exception')),

  maersk\_carrier\_booking\_reference text,

  maersk\_transport\_document\_reference text,

  maersk\_equipment\_reference text,

  freight\_cost numeric,

  created\_at timestamptz default now(),

  updated\_at timestamptz default now()

);

create table shipment\_parcels ( \-- many-to-many consolidation

  shipment\_id uuid references shipments(id) on delete cascade,

  parcel\_id uuid references parcels(id) on delete cascade,

  primary key (shipment\_id, parcel\_id)

);

create table tracking\_events ( \-- cached Maersk events \+ manual milestones

  id uuid primary key default gen\_random\_uuid(),

  shipment\_id uuid references shipments(id) on delete cascade,

  source text not null check (source in ('maersk','manual')),

  event\_type text, \-- SHIPMENT / TRANSPORT / EQUIPMENT or custom manual label

  description text,

  location text,

  event\_datetime timestamptz,

  raw\_payload jsonb, \-- store full Maersk event object for audit

  created\_at timestamptz default now()

);

**Row Level Security (RLS):**

- `profiles`, `parcels`, `shipments`: customers can `select`/`insert` only their own rows (`customer_id = auth.uid()`); `warehouse_staff`/`admin` roles (checked via a `profiles.role` lookup or custom claim) can read/update all rows.  
- `tracking_events`: read-only for customers, scoped via a join to `shipments.customer_id = auth.uid()`. Only server-side service-role key writes to this table (from the Maersk sync job).

## 5\. Pages & Features

**Public site:**

- **Home** — hero with prominent tracking search box (accepts an STC tracking number OR warehouse code \+ local tracking number), services overview (Air Freight / Sea Freight / Consolidation), "How It Works" 4-step visual (Get your code → Shop & pre-alert → Warehouse consolidates → Ship & track), pricing/quote request form, testimonials, contact/WhatsApp.  
- **Track Shipment** (`/track`) — single input for the internal STC tracking number; server-side route resolves it to a shipment, fetches cached/live events, renders a vertical timeline with status, location, ETA. No outbound links to Maersk.  
- **Services** pages for Air Freight and Sea Freight (static content).  
- **Sign up / Log in** (Supabase Auth — email/password, optionally phone OTP for the China market).

**Customer dashboard** (`/dashboard`, auth-protected):

- My Warehouse Code (with copy button \+ instructions to give this to suppliers/staff).  
- Submit Pre-Alert form (local tracking number \+ item details).  
- My Parcels table (status: pending / arrived / consolidated).  
- Request Shipment flow (select arrived parcels → destination → mode → submit).  
- My Shipments table with tracking timeline per shipment.

**Staff/Admin panel** (`/admin`, role-protected):

- Incoming pre-alerts queue; search/mark parcels as arrived; upload photos/weight.  
- Shipment builder: consolidate parcels into a shipment, generate STC tracking number, attach Maersk references, set freight cost.  
- Manual milestone entry (for air freight and Maersk-fallback cases).  
- Basic CRM: customer list, parcel/shipment counts.

## 6\. Tracking Resolution Logic (server-side API route, e.g. `/api/track/[stcNumber]`)

1. Look up `shipments` by `stc_tracking_number`.  
2. If `maersk_transport_document_reference` or `maersk_carrier_booking_reference` is set:  
   - Check `tracking_events` cache; if freshest event is \< 30 min old, serve from cache.  
   - Otherwise call Maersk `/events` server-side with the stored reference and `eventType=SHIPMENT,TRANSPORT,EQUIPMENT`.  
   - On success: upsert new events into `tracking_events` (source \= `maersk`), return merged timeline.  
   - On 404/error: log it, fall back to any `manual` events already in `tracking_events`, and show a generic "in transit — next update pending" state rather than an error.  
3. If no Maersk reference is attached (air freight or not yet booked with Maersk), serve only `manual` events entered by staff.  
4. Always merge `maersk` \+ `manual` events into one chronological timeline in the UI, so the customer sees one consistent tracking experience regardless of source.

## 7\. Non-Functional Requirements

- All Maersk credentials and the Supabase **service role key** live only in Vercel server-side env vars (`MAERSK_CONSUMER_KEY`, `MAERSK_CONSUMER_SECRET`, `SUPABASE_SERVICE_ROLE_KEY`, etc.) — never in client bundles or `NEXT_PUBLIC_*` vars.  
- Use Supabase Auth for session management; use RLS as the primary authorization layer, not just app-level checks.  
- Mobile-first responsive design; the tracking widget should be usable on a phone (primary device for many customers in this market).  
- Warehouse codes and STC tracking numbers should be short, human-readable, and easy to read over the phone/WhatsApp to warehouse staff.  
- Add basic rate-limiting/captcha on the public tracking form to prevent scraping/enumeration of tracking numbers.  
- Set up a Vercel Cron job (e.g. `/api/cron/sync-maersk`) to periodically refresh events for all active shipments with a Maersk reference, so customers get near-real-time data without triggering a live API call on every page load.
